## Example 1: SLHS Voice Agent

**Path:** `../slhs-voice-agent/`  
**Live URL:** `https://devex-orchestrator-dev.greenbay-9ec52bc2.eastus2.azurecontainerapps.io`

### What It Is
Voice-enabled patient information system for St. Luke's Hospital Network. Cross-browser speech recognition with auto-retry, 5 synthetic patients with full clinical data (medications, vitals, labs), appointment scheduling with 5 doctors, multi-turn conversation context.

### Key Features
- Voice chat with exponential backoff retry (3 attempts)
- 10+ intent categories (patient lookup, medications, scheduling, emergency)
- HTML card rendering for structured clinical data
- HIPAA-aware data handling (synthetic PHI only)
- Session management with multi-turn context

### Technologies
- FastAPI + Uvicorn (2 workers)
- Azure Container Apps
- Log Analytics + Application Insights
- Managed Identity
- Key Vault

### Deployment
```bash
cd slhs-voice-agent
az deployment group create --resource-group rg-voice-agent-dev --template-file infra/bicep/main.bicep
az acr build --registry <ACR> --image voice-agent:v3.0.1 --no-logs src/app/
```

### Intent File
See `examples/intent.md` (v1) and `examples/intent.v2.md` (upgrade example)

### Scorecard
- **135/135 total** (100 core + 35 bonus)
- **26/26 WAF principles** at 100%
- **24/25 governance policies** passing (1 warning on network isolation)

## Example 2: Legal Contract Review and Redlining AI ⭐ NEW

**Path:** `../contract-review/`  
**Deploy:** 3 steps (see contract-review/README.md)

### What It Is
AI-powered contract analysis for St. Luke's Hospital Network legal team. Upload vendor contracts (PDF/DOCX), extract text via Document Intelligence, analyze with GPT-4-1 against healthcare risk framework, generate redline suggestions with protective alternative clauses.

### Key Features
- Document Intelligence text extraction (50MB, 200 pages supported)
- GPT-4-1 risk analysis across 6 categories:
  - Liability limitations
  - Indemnification
  - HIPAA compliance (BAA requirements)
  - Data security & breach notification
  - Insurance requirements
  - Termination clauses
- Standard clause library (20+ protective clauses)
- Redline export to DOCX with track changes
- RBAC with 4 roles (Legal.Admin, Legal.Reviewer, Procurement.Officer, Department.Head)

### Technologies
- FastAPI + Pydantic
- Azure Container Apps
- Blob Storage (contract files)
- Cosmos DB (analysis results, clause library)
- Document Intelligence (OCR + layout)
- Azure OpenAI/Foundry (GPT-4-1)
- Managed Identity + Key Vault
- Log Analytics

### Deployment (Option A: GitHub Actions)

## How These Were Built

Both examples were generated using the same orchestrator pipeline:

```
1. Write comprehensive intent file (9 enterprise sections)
   ↓
2. devex scaffold --file intent.md -o ./app-name
   ↓
3. Review governance report + WAF assessment
   ↓
4. Deploy to Azure (bicep → acr → container app)
   ↓
5. Iterate based on improvement suggestions
```

**Total authoring time per example:** ~20 minutes (intent file) + 2 minutes (orchestrator execution) + 10 minutes (Azure deployment)

## Lessons Learned

### Voice Agent (v1.0.0 → v3.0.1)
- **v1.0.0** -- Initial scaffold, basic patient lookup
- **v2.0.0** -- Added HTML cards, multi-turn context, appointment scheduling
- **v3.0.0** -- Enterprise rewrite with 5 patients, cross-browser voice
- **v3.0.1** -- Voice network error fix with auto-retry

**Key Insight:** The orchestrator scaffold provided the foundation, but iterative refinement based on user testing drove v2.0+ features.

### Contract Review (v1.0.0)
- **v1.0.0** -- Generated from comprehensive intent file on first try
- Governance report: 25/25 policies passing
- WAF assessment: 26/26 principles covered
- Improvement suggestions: 10 enhancements for v2 (Redis cache, FHIR integration, VNet isolation)

**Key Insight:** Investing time in the 9 enterprise sections (problem, goals, users, requirements, scalability, security, performance, integration, acceptance) results in a more complete v1 scaffold.

## Contributing New Examples

Want to add a new example? Follow these steps:

1. **Create intent file** -- Use `devex init` or copy `examples/intent.md` as template
2. **Fill 9 enterprise sections** -- Be comprehensive (see contract-review-intent.md as model)
3. **Generate scaffold** -- `devex scaffold --file intent.md -o ./new-example`
4. **Deploy to Azure** -- Verify it works end-to-end
5. **Document deployment** -- Add quick deploy section to README (3-5 steps max)
6. **Test** -- Run `pytest tests/ -v` and ensure all pass
7. **Add to this README** -- Include comparison row, lessons learned
8. **Commit** -- No credentials, use managed identity for all auth
